# Some Cleaner 개발 과정 및 현재 코드 분석 정리

## 작성 기준

- 작성일: 2026-04-17
- 기준 자료: 현재 `main` 브랜치의 Git 히스토리, `CLAUDE.md`, `POLICY.md`, 현재 `src/` 코드 구조
- 목적: 지금까지의 개발 흐름을 빠르게 복기하고, 이후 유지보수 시 바로 참고할 수 있는 분석 메모를 남긴다

## 프로젝트 목표

Some Cleaner는 사내 Linux 서버의 디스크 정리 작업을 웹에서 등록하고 추적하는 프론트엔드다.

- 대시보드에서 디스크 사용량을 확인한다.
- 디스크별 삭제 작업을 등록한다.
- 마지막 실행값을 불러와 재입력 비용을 줄인다.
- 처리 내역에서 현재 상태를 확인한다.

핵심 도메인 모델은 "디스크 마스터 설정"보다 "삭제 실행 1건 = 설정 스냅샷 1건"에 가깝다. 이 판단은 현재 화면 구조와 API 설계 전반에 그대로 반영돼 있다.

## 개발 과정 요약

### 1. 초기 기반 구축

초기 커밋(`12e44c0`)에서 Vite + React + TypeScript 기반 프로젝트를 만들고, Tailwind, 공통 유틸, MSW 초안을 올렸다. 이 시점에 이미 사외 개발과 mock 기반 검증을 전제로 구조를 잡았다.

이후 `b734b09`, `1bf4c80`, `daed842`에서 다음 성격의 기반 보강이 있었다.

- MSW service worker 경로 보정
- mock 초기화 실패 시에도 앱 렌더 유지
- 환경변수 이름 정리 (`VITE_API_URL`)

즉, 초반부터 "실제 백엔드 없이도 앱이 멈추지 않게 개발한다"는 방향이 분명했다.

### 2. 배포/이전 환경 정리

`46f09c4`, `f748445`, `14e3f76`, `499ca1f`에서는 사내 이전과 검증 흐름을 정리했다.

- 개발환경 안내 문서 정비
- PowerShell 기반 내부 반입 스크립트 추가 및 교체
- 인트라넷 이전 테스트용 진입 화면 조정

이 프로젝트는 일반적인 인터넷 배포보다 "사외 개발 → 사내 반입 → Jenkins/K8s 배포"가 더 중요한 제약이므로, 문서와 스크립트가 초기에 함께 만들어진 점이 특징이다.

### 3. 핵심 앱 골격 구현

`2aa85b0`에서 로그인, 라우팅, 레이아웃, 목록/등록 화면의 초안을 한 번에 올렸고, `928bbc8`, `2f7f41b`에서 라우팅 안정성을 보강했다.

이 단계에서 자리 잡은 핵심 구조는 다음과 같다.

- `react-router-dom` 기반 라우팅
- `zustand` 기반 auth/store 분리
- `axios` 공통 클라이언트 + Authorization 헤더 주입
- 로그인 후 보호 라우트 접근

초기 명칭은 `scripts` 중심이었지만, 이후 도메인이 더 명확해지면서 `disk`와 `process` 중심 구조로 정리됐다.

### 4. 디스크/삭제 프로세스 도메인으로 전환

`9da1952`는 현재 제품 형태를 결정한 큰 전환점이다.

- 대시보드 추가
- 디스크 카드 UI 추가
- 삭제 등록 페이지 추가
- 처리 내역 페이지 추가
- API/store/type 명칭을 `script`에서 `process`/`disk` 중심으로 재편

이 시점부터 앱은 "삭제 스크립트 목록 도구"보다 "디스크 정리 운영 화면"으로 성격이 바뀌었다.

### 5. UI 확장 후 단순화

`393a537`, `ed8fad2`에서는 카드/리스트 토글, 확인 모달, 디스크 설정 API 연동 같은 기능이 추가됐다. 당시 방향은 "디스크 설정"과 "삭제 실행"을 분리해서 다루는 쪽이었다.

하지만 최종적으로 `4b5e5c2`에서 흐름이 다시 단순화됐다.

- 별도 디스크 설정 저장 흐름 제거
- `latest-config` 기반 prefill 유지
- 삭제 등록을 바로 실행하는 폼으로 회귀

현재 구조는 복잡한 설정 관리보다 운영자가 빠르게 재실행하는 시나리오에 더 맞춰져 있다.

### 6. PathInput 입력 UX 이슈 집중 수정

`bf1ab7b`부터 `16d7c15`까지의 커밋들은 대부분 `PathInput`의 Enter 처리와 포커스 이동 문제를 다뤘다.

중간에 시도된 접근은 다음과 같다.

- Enter bubbling 차단
- uncontrolled input 전환 시도
- 부모 form 제출 차단
- native capture-phase listener 사용
- form 제거
- `useLayoutEffect`로 포커스 복원

최종 결과물은 현재 코드처럼 아래 조합으로 정리됐다.

- 컴포넌트 내부 `form`에서만 submit 처리
- 조합 중 입력(`IME composition`) 방어
- `onKeyDownCapture`에서 Enter 전파 차단
- 추가 직후 다시 입력창에 포커스를 주는 흐름 유지

이 히스토리는 단순한 UI 버그가 아니라, 실제 운영 입력 폼에서 반복 사용성을 얼마나 중요하게 본 프로젝트인지 보여준다.

## 현재 코드 구조 분석

### 라우팅과 인증

현재 라우트는 `login`, `dashboard`, `process/new`, `history` 네 축으로 단순하다.

- `/login`: mock 또는 SSO 진입
- `/dashboard`: 디스크 목록과 삭제 등록 진입
- `/process/new?diskId=:id`: 삭제 작업 생성
- `/history`: 생성된 작업 상태 조회

`RequireAuth`는 `useAuthStore.isAuthenticated`만 본다. 인증 상태 판단은 빠르지만, 토큰만 복구되고 사용자 정보는 복구되지 않는 구조라 새로고침 직후 헤더 사용자명이 비어 있을 수 있다.

### API와 상태관리

데이터 접근 계층은 비교적 얇다.

- `src/api/client.ts`: 공통 axios 인스턴스
- `src/api/auth.ts`: 로그인, 내 정보 조회
- `src/api/disks.ts`: 디스크 목록 조회
- `src/api/process.ts`: 프로세스 목록 조회, 최신 설정 조회, 생성

상태는 `zustand`로 기능별 분리돼 있다.

- `useAuthStore`
- `useDiskStore`
- `useProcessStore`

현재는 React Query 같은 캐시 계층 없이 "페이지 진입 시 fetch" 패턴을 직접 구현하고 있다. 규모가 작은 동안에는 단순하지만, 실패/재시도/동기화 정책은 각 store가 직접 책임져야 한다.

### 화면별 책임

`DashboardPage`

- 디스크 목록 조회
- 카드/리스트 뷰 토글
- 선택 디스크의 삭제 등록 화면 이동

`DeleteProcessPage`

- `diskId` 파라미터 해석
- 디스크 목록이 없으면 먼저 로드
- `GET /api/process/latest-config/:diskId/`로 마지막 실행값 prefill
- 내부 경로, 외부 경로, To, CC 검증 후 생성

`HistoryPage`

- 프로세스 목록 조회
- 상태 badge와 생성 시각 표시

### 입력 컴포넌트 전략

`PathInput`은 현재 앱에서 가장 중요한 재사용 입력 컴포넌트다.

- 단일 Enter 입력 추가
- 여러 줄 붙여넣기 일괄 추가
- 중복 제거
- 개별 삭제 / 전체 삭제

삭제 등록 화면의 생산성은 사실상 이 컴포넌트 품질에 달려 있다. 그래서 Git 히스토리에서도 가장 많은 수정이 이 컴포넌트에 집중됐다.

### Mock 전략

MSW는 단순한 개발 편의 기능을 넘어서, 폐쇄망 환경 때문에 필수적인 개발 기반 역할을 한다.

현재 mock은 다음 API를 흉내 낸다.

- `POST /api/auth/login`
- `GET /api/auth/me`
- `GET /api/disks/`
- `GET /api/process/latest-config/:diskId/`
- `GET /api/process/`
- `POST /api/process/`

Mock 데이터가 `latest-config`와 이력 화면까지 포함하고 있어, 백엔드 없이도 핵심 플로우 검증이 가능하다.

## 현재 설계의 장점

### 1. 운영 플로우에 맞게 복잡도를 줄였다

디스크별 영구 설정 화면을 별도로 두지 않고, 마지막 실행값을 재사용하는 방식으로 단순화했다. 운영자가 같은 형식의 작업을 반복 등록하는 상황에는 이 구조가 더 빠르다.

### 2. 화면 책임이 비교적 명확하다

대시보드, 등록, 이력 세 화면의 역할이 겹치지 않는다. 소규모 팀이 유지하기 좋은 구조다.

### 3. mock과 실제 API 경계가 분리돼 있다

실제 화면 코드가 mock 여부를 거의 의식하지 않고, 환경변수와 API 계층에서 전환된다. 사외 개발 환경을 감안하면 합리적인 분리다.

### 4. 입력 UX에 대한 학습이 코드에 반영돼 있다

`PathInput`의 현재 구현은 단순히 동작만 하는 상태가 아니라, 실제 Enter/포커스/붙여넣기 문제를 여러 번 겪고 정리한 결과다.

## 현재 코드 기준 리스크와 보완 포인트

### 1. store 로딩 상태가 실패 시 복구되지 않을 수 있다

`useDiskStore.fetchDisks`, `useProcessStore.fetchProcesses`는 요청 전에 `isLoading`을 `true`로 올리지만, 실패 시 `finally`가 없어 로딩 상태가 내려오지 않을 수 있다. 네트워크 오류가 실제로 발생하면 화면이 계속 로딩처럼 보일 가능성이 있다.

### 2. 인증 부트스트랩이 없다

`useAuthStore`는 `localStorage`의 토큰만 복구하고 `user`는 복구하지 않는다. `getMe()`를 앱 시작 시 한 번 호출하는 부트스트랩이 없어서, 새로고침 후 인증 여부와 사용자 표시 상태가 어긋날 수 있다.

### 3. 이력 화면은 실시간성이 없다

`HistoryPage`는 최초 진입 시 한 번만 불러온다. 실제 배치 실행 상태가 `pending -> running -> completed`로 바뀌어도 새로고침 전에는 반영되지 않는다.

### 4. 오류 처리 경험이 아직 얇다

현재는 대부분의 실패를 페이지 내부 문구로만 보여준다. 운영 도구 특성상 API 실패, 권한 문제, SSO 만료, 404/500 구분 등이 더 명확해질 필요가 있다.

### 5. 삭제 등록 화면은 두 종류의 선행 데이터에 의존한다

`DeleteProcessPage`는 디스크 목록과 latest config를 별도로 본다. 동작은 가능하지만, 페이지 자체의 로딩 상태를 한 번에 관리하지 않아 사용자 입장에서는 "디스크 정보 로딩"과 "기본값 로딩"이 분리돼 보일 수 있다.

## 다음 작업 후보

우선순위를 기준으로 보면 다음 순서가 자연스럽다.

1. store fetch 함수에 `try/catch/finally`를 넣어 실패 시 로딩 상태와 에러 상태를 정리
2. 앱 시작 시 `getMe()` 기반 인증 부트스트랩 추가
3. `HistoryPage`에 polling 또는 수동 새로고침 버튼 추가
4. 에러 표시 체계를 공통 컴포넌트 또는 패턴으로 정리
5. 실제 Django 응답 스키마와 mock 응답 스키마 차이를 문서화

## 한 줄 결론

지금까지의 개발은 "폐쇄망 제약이 있는 운영용 도구를 최대한 단순한 화면 구조로 빠르게 굴릴 수 있게 만드는 과정"이었다. 현재 코드는 이미 핵심 사용자 흐름을 갖췄고, 다음 단계의 핵심은 기능 추가보다 실패 처리와 상태 동기화 안정성을 높이는 쪽에 있다.

## SSO 리다이렉트 메모

- 운영/K8s 환경에서는 프론트가 `fromPath`, `toPath` 쿠키를 `.samsungds.net` domain으로 심고 Django SSO 엔드포인트로 이동한다.
- Django는 ADS SSO 이후 `request.COOKIES.get('toPath')`를 읽어 최종 경로로 리다이렉트한다.
- 로컬 `localhost`에서는 `.samsungds.net` 쿠키를 설정할 수 없으므로 real SSO를 지원하지 않고 mock 로그인을 사용한다.
